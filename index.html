PDF 1: Fundamentos de Bases de Datos

Introducción a las Bases de Datos (BD)

¿Qué es una Base de Datos (BD)? Es una colección de datos interrelacionados y relevantes para una empresa u organización. Piensa en ella como un archivador digital súper organizado.

¿Qué es un Sistema Gestor de Bases de Datos (SGBD)? Es el software (conjunto de programas) que nos permite crear, acceder y gestionar esa base de datos. Es como el bibliotecario que sabe dónde está cada libro y cómo encontrarlo.

Objetivo Principal de un SGBD: Almacenar y recuperar información de una BD de forma práctica y eficiente. Queremos que sea rápido y fácil encontrar lo que buscamos.

Importancia: Los SGBD manejan grandes cantidades de información, definen cómo se guarda (estructuras) y cómo se manipula.

Aplicaciones Comunes

  • Información empresarial: Inventarios, ventas, clientes.
  • Banca y finanzas: Cuentas, transacciones.
  • Universidades: Estudiantes, calificaciones, cursos.
  • Líneas Aéreas: Reservas, vuelos.
  • Telecomunicaciones: Clientes, llamadas, planes.

Conceptos Clave

Dato: Una representación simbólica (número, letra, etc.) de información. Se usa para tomar decisiones. Ejemplo: tu edad es un dato (25), tu nombre es un dato ("Ana").

Clasificación de Tipos de Datos:

  • Datos estáticos: Su tamaño en memoria no cambia durante la ejecución del programa (ej: un número entero siempre ocupa lo mismo).
  • Datos dinámicos: Su tamaño puede variar mientras el programa se ejecuta (ej: una lista de nombres que puede crecer).
  • Datos simples: Los básicos. Como números (entero), letras (carácter), números con decimales (real).
  • Datos estructurados: Colecciones de elementos (ej: una lista de alumnos, donde cada alumno es un elemento).

Actores en una Base de Datos:

Son las personas que interactúan con la BD:

  • Administrador de Bases de Datos (ABD): El "jefe" de la BD, se encarga de que todo funcione bien, seguridad, copias de respaldo.
  • Diseñador de Bases de Datos: El arquitecto, decide cómo se va a organizar la información.
  • Usuario final: Quien usa la BD para consultar, actualizar o generar informes (ej: un cajero en un banco).
  • Analista de sistemas y programador de aplicaciones: Crean los programas que usan la BD.

Modelos de Bases de Datos

Un modelo de datos es como un plano o un conjunto de reglas para describir los datos, cómo se relacionan y qué restricciones tienen.

Componentes de un Modelo de Datos:

  • Estructuras de datos: Define qué tipos de datos habrá y cómo se conectan entre sí.
  • Restricciones de integridad: Reglas para que los datos sean correctos y reflejen la realidad (ej: un DNI debe ser único).

Tipos de Modelos de BD (¡Importante!):

  • Relacional: El más común. Organiza los datos en tablas (como hojas de cálculo). Cada tabla tiene filas (registros o tuplas) y columnas (campos o atributos). Se usa SQL para consultarlo.
  • Entidad-Relación (E-R): Un modelo conceptual para diseñar BD. Se basa en:
    • Entidades: "Cosas" u "objetos" del mundo real (ej: Alumno, Curso).
    • Atributos: Características de las entidades (ej: Nombre del alumno, Código del curso).
    • Relaciones: Cómo se conectan las entidades (ej: un Alumno "se inscribe en" un Curso).
  • Basado en objetos (Orientado a Objetos): Extiende el E-R. Piensa en los datos como "objetos" que tienen datos y funciones. Conceptos clave: Encapsulación (proteger datos), Herencia (objetos heredan características), Polimorfismo (una misma operación se comporta diferente según el objeto).
  • Semiestructurados: Permite que datos del mismo tipo tengan diferentes atributos. XML es un ejemplo. Es más flexible.
  • Jerárquicos: Como un árbol genealógico. Un nodo "padre" puede tener varios "hijos", pero cada "hijo" solo un "padre". Antiguo, puede generar redundancia.
  • De Red: Similar al jerárquico, pero un "hijo" puede tener varios "padres". Más flexible que el jerárquico para evitar redundancia, pero complejo de administrar.

Dato curioso: Los modelos Jerárquico y De Red son más antiguos y aunque potentes en su momento, son menos comunes hoy en día excepto en sistemas "legacy" (viejitos pero que siguen funcionando).

PDF 2: Características, Funciones y Componentes

Características de una BD (¡Muy Importante!)

  1. Almacenamiento estructurado:
    • Datos organizados en tablas (BD Relacionales como MySQL, PostgreSQL) o en otros formatos (NoSQL: documentos, grafos).
    • Tablas con registros (filas) y campos (columnas) con tipos de datos definidos (ej: texto, número, fecha).
  2. Integridad de los datos:
    • Reglas de validación: Aseguran que los datos cumplan condiciones (ej: campo obligatorio, formato de email correcto).
    • Claves primarias y foráneas: Evitan duplicados y mantienen relaciones correctas entre tablas. ¡Fundamentales!
    • Transacciones ACID (en BD Relacionales): Atomicidad (todo o nada), Consistencia (datos válidos), Aislamiento (transacciones no se interfieren), Durabilidad (cambios persisten). Piensa en una transferencia bancaria: o se hace todo (quitar de una cuenta y poner en otra) o no se hace nada.
  3. Acceso concurrente y control de usuarios:
    • Varios usuarios pueden acceder a la vez.
    • Sistemas de permisos (roles y privilegios) para seguridad (ej: un usuario solo puede leer, otro puede modificar).
  4. Lenguaje de consulta (SQL o alternativas):
    • SQL (Structured Query Language): Para insertar, modificar, eliminar y consultar datos (ej: SELECT, INSERT, UPDATE, DELETE). El idioma universal de las BD relacionales.
    • NoSQL: Usan lenguajes propios como MQL (MongoDB Query Language) o APIs específicas.
  5. Escalabilidad y rendimiento:
    • Escalabilidad vertical: Aumentar capacidad del servidor (más RAM, CPU más potente).
    • Escalabilidad horizontal: Distribuir la carga en múltiples servidores (común en NoSQL).
    • Índices: Como el índice de un libro, ayudan a encontrar datos más rápido.
  6. Seguridad:
    • Cifrado de datos sensibles: Proteger información importante.
    • Copias de seguridad (backups) y recuperación ante fallos. ¡Vital!
    • Auditoría de accesos: Saber quién hizo qué y cuándo.
  7. Independencia entre datos y aplicaciones:
    • Cambios en la estructura de la BD no deberían afectar (idealmente) a las aplicaciones que la usan.
  8. Soporte para grandes volúmenes de datos (Big Data):
    • Algunas BD están optimizadas para manejar cantidades masivas de información (Petabytes). Ej: Hadoop, Cassandra.

Funciones de una BD

Una BD hace muchas cosas, entre ellas:

  • Almacenar datos de forma estructurada.
  • Recuperar datos (Consultas).
  • Permitir insertar, actualizar y eliminar datos.
  • Mantener la integridad de los datos.
  • Controlar que varios usuarios puedan acceder sin problemas (Concurrencia).
  • Proveer seguridad y control de acceso.
  • Realizar respaldos y permitir la recuperación.
  • Optimizar el rendimiento para que sea rápida.
  • Soportar transacciones (ACID).
  • Integrarse con aplicaciones.
  • Ser escalable (crecer según se necesite).
  • Permitir análisis y generar reportes.

Componentes de un Sistema de Base de Datos (SBD)

Un SBD no es solo el software, es todo el conjunto:

  1. Datos: La información en sí. El corazón del sistema.
  2. Hardware: Servidores, discos duros (SSD para velocidad), etc.
  3. Software: El SGBD (ej: PostgreSQL, MySQL, MongoDB), sistema operativo, aplicaciones.
  4. Usuarios: Administradores, diseñadores, programadores, usuarios finales.
  5. Lenguajes de Base de Datos: SQL, MQL, etc.
  6. Procedimientos y Protocolos: Reglas y pasos para usar y administrar el sistema (ej: cómo hacer backups, cómo dar permisos).

Ejemplo de un SBD en un Hospital:

  • Datos: Historias médicas de pacientes (en una tabla "Pacientes").
  • Hardware: Un servidor con discos SSD rápidos.
  • Software: PostgreSQL (SGBD) + una interfaz web para los médicos.
  • Usuarios: Médicos (consultan), Administradores (hacen backups).
  • Lenguaje: SQL para generar reportes (ej: SELECT * FROM Pacientes WHERE edad > 60).

Componentes de un SBD ajustados a las necesidades de una organización

Crear un SBD para una empresa implica varios pasos:

  1. Análisis de Requerimientos:
    • Objetivo: Entender qué datos maneja la organización y cómo los usa.
    • Métodos: Entrevistas (con ventas, logística, RRHH), revisar documentos (facturas, inventarios).
    • Preguntas clave: ¿Qué datos se generan? ¿Qué volumen? ¿Acceso en tiempo real o análisis histórico?
    • Salida: Lista de entidades (futuras tablas) y operaciones importantes.
  2. Selección del Modelo de Datos:
    • Opción A: Relacional (SQL): Ideal para datos estructurados con relaciones complejas (ej: un ERP bancario), necesidad de transacciones ACID (sistemas contables). Ejemplos: PostgreSQL, MySQL, Oracle.
    • Opción B: NoSQL: Ideal para datos semi-estructurados o no estructurados (redes sociales), escalabilidad horizontal (apps móviles con millones de usuarios). Tipos: Documental (MongoDB para perfiles de usuario), Clave-Valor (Redis para caché de sesiones), Grafos (Neo4j para recomendaciones).
    • Criterios de decisión: Estructura de los datos, volumen y velocidad de crecimiento, si se necesita más consistencia (SQL) o más disponibilidad (NoSQL).
  3. Diseño de la Arquitectura:
    • Monolítica: BD centralizada (para PYMEs con bajo tráfico).
    • Cliente-Servidor: Aplicaciones acceden a un servidor de BD. El estándar.
    • Distribuida: Fragmentos de BD en múltiples nodos (ej: bancos con sucursales globales).
    • Híbrida: Combina SQL + NoSQL (ej: datos transaccionales en SQL y logs en MongoDB).
    • Herramientas de modelado: Diagramas E-R (para SQL), esquemas JSON/BSON (para NoSQL).
  4. Selección de Hardware y Software:
    • Hardware:
      • Servidores: On-premise (control total, para empresas con regulaciones estrictas) o Cloud (AWS RDS, Azure SQL - escalabilidad y mantenimiento gestionado).
      • Almacenamiento: SSDs (alta velocidad), HDDs (datos históricos).
    • Software (DBMS):
      • Criterios: Costo (open-source vs. licencias), soporte para replicación y backups, compatibilidad con lenguajes de programación (Python, Java).
  5. Definición de Roles y Protocolos:
    • Usuarios y permisos:
      • Roles: DBA (acceso total), Desarrolladores (permisos DML como SELECT, INSERT), Analistas (solo consultas).
    • Herramientas de seguridad: Cifrado (TLS para conexiones, AES-256 para datos en reposo), Auditoría (logs de accesos).
    • Procedimientos operativos: Backup (frecuencia, método incremental/completo), Recuperación ante desastres (Plan con RTO - tiempo para recuperar, y RPO - cuántos datos se pueden perder).
  6. Pruebas y Optimización:
    • Pruebas de carga: Simular miles de usuarios (ej: con JMeter). Ajustar índices y particiones.
    • Monitorización continua: Herramientas (Prometheus para métricas, Elasticsearch para logs). Métricas clave (tiempo de respuesta de consultas, uso de CPU/memoria).

PDF 3: Modelo Entidad-Relación (E-R)

Este PDF profundiza en el modelo E-R, crucial para diseñar bases de datos antes de construirlas.

Modelo Entidad-Relación (E-R) - Repaso y Profundización

El modelo E-R nos ayuda a ver el mundo real y traducirlo a un diseño de BD. Se compone de:

  • Entidades: Objetos básicos distinguibles. Pueden ser concretas (una persona) o abstractas (una fecha). Un "conjunto de entidades" agrupa entidades del mismo tipo (ej: todos los ALUMNOS de la UTM).
  • Atributos: Características que describen una entidad (ej: para Alumno: Nombre, Expediente, Carrera). Cada atributo tiene un "dominio" (valores permitidos, ej: para 'Edad', el dominio son números enteros positivos).
  • Relaciones: Asociaciones entre dos o más entidades (ej: un ALUMNO "cursa" una MATERIA).

Grado de una Relación:

Indica cuántas entidades participan en la relación.

  • Grado 1 (Unitaria): Una entidad se relaciona consigo misma (ej: un empleado es jefe de otros empleados).
  • Grado 2 (Binaria): Dos entidades distintas se relacionan (ej: un PROVEEDOR "suministra" ARTÍCULOS). Es la más común.
  • Grado 3 (Ternaria): Tres o más entidades se relacionan (ej: un CLIENTE usa una CUENTA en una SUCURSAL).

Limitantes de Mapeo (Cardinalidad y Dependencia)

Son reglas para que los datos sean realistas.

1. Tipos de Relaciones (Cardinalidad):

Define cuántas instancias de una entidad B se pueden relacionar con una instancia de una entidad A (y viceversa). ¡Esto es súper importante!

  • Uno a Uno (1:1): Una instancia de A se relaciona con UNA SOLA de B, y una de B con UNA SOLA de A.
    • Ejemplos: CONTRIBUYENTE – RFC (cada persona un RFC, cada RFC una persona), ALUMNO – No. MATRÍCULA.
  • Uno a Muchos (1:N): Una instancia de A se relaciona con VARIAS de B, pero una de B se relaciona con UNA SOLA de A.
    • Ejemplos: CLIENTE – CUENTAS BANCARIAS (un cliente puede tener varias cuentas, una cuenta pertenece a un solo cliente), PADRE – HIJOS.
  • Muchos a Uno (N:1): Varias instancias de A se relacionan con UNA SOLA de B. Es el inverso de 1:N.
    • Ejemplos: EMPLEADOS – EMPRESA (varios empleados trabajan en una empresa), ALUMNOS – ESCUELA.
  • Muchos a Muchos (N:N): Una instancia de A se relaciona con VARIAS de B, y una de B también se relaciona con VARIAS de A.
    • Ejemplos: ESTUDIANTES – MATERIAS (un estudiante cursa varias materias, una materia es cursada por varios estudiantes), PROYECTOS – ARQUITECTOS.

Nota: La cardinalidad depende del "punto de vista" o las reglas del negocio. Por ejemplo, en un sistema un empleado podría estar asignado a un solo proyecto (1:1 con proyecto) o a varios (1:N con proyecto).

Formas de representar la cardinalidad: Aparte de 1, N, M, se usa notación como (0,1) [cero o uno], (1,1) [exactamente uno], (0,n) [cero o muchos], (1,n) [uno o muchos]. El primer número es la participación mínima, el segundo la máxima.

2. Dependencia de Existencia:

Si la entidad A "depende" de la entidad B para existir, entonces A es "dependiente de existencia" de B. Si borras B, A también debe borrarse. B es la entidad "Dominante", A es la "Subordinada".

Llaves (Claves)

  • Llave Primaria (Primary Key - PK): Atributo o conjunto de atributos que identifica de forma ÚNICA a cada instancia (fila) en una entidad (tabla). Ejemplo: No. Matrícula para Alumno. Se suele subrayar en los diagramas.
  • Llaves Secundarias (Candidate Keys): Otros atributos que también podrían ser PK porque son únicos (ej: CURP, RFC para una Persona, si la PK es un ID numérico).

Tipos de Atributos (¡También Importante!)

  • Simples: No se pueden dividir más (ej: Edad, Género).
  • Compuestos: Se pueden dividir en partes más pequeñas con significado propio (ej: Dirección se compone de Calle, Número, Ciudad, CP; Nombre completo se compone de Nombre, Ap. Paterno, Ap. Materno).
  • Almacenados: Su valor se guarda directamente en la BD (ej: Fecha de Nacimiento).
  • Derivados (o Calculados): Su valor se calcula a partir de otros atributos y NO se guarda directamente (ej: Edad, se calcula desde Fecha de Nacimiento y la fecha actual). Se representan con un óvalo punteado.
  • Monovaluados (Univaluados): Tienen un solo valor para cada entidad (ej: una persona tiene un solo CURP).
  • Multivaluados: Pueden tener múltiples valores para una misma entidad (ej: una persona puede tener varios Números de Teléfono, varios hobbies). Se representan con un doble óvalo. A menudo indican la necesidad de una tabla separada en el modelo relacional.
  • Opcionales (Nulos): Pueden no tener valor. Un valor "nulo" significa que el valor es desconocido o no aplica (ej: Segundo Apellido puede ser nulo, Email).

Atributos en las Relaciones

A veces, una relación puede tener sus propios atributos. Por ejemplo, en la relación "un ALUMNO se INSCRIBE en una MATERIA", la "Fecha de Inscripción" o la "Calificación" son atributos de esa inscripción específica, no solo del alumno ni solo de la materia.

Cuando una relación N:N tiene atributos, o incluso a veces en relaciones 1:N, esto a menudo se resuelve convirtiendo la relación en una nueva entidad (llamada entidad asociativa o tabla intermedia) en el modelo relacional. Esta nueva entidad tendrá como llaves foráneas las llaves primarias de las entidades que relaciona, y sus propios atributos.

Ejemplo: Si PERSONA "ocupa" HABITACIÓN con atributos Fecha_Ingreso y Fecha_Salida, se podría crear una entidad HOSPEDAJE con esos atributos y relacionada con PERSONA y HABITACIÓN.

PDF 4: Normalización y Modelo Relacional

Transformación del Modelo E-R al Modelo Relacional

El modelo E-R es un diseño. El modelo Relacional es cómo se implementa usando tablas. Aquí las reglas clave para "traducir":

  • Cada Entidad se convierte en una Tabla. Los atributos de la entidad son las columnas de la tabla. La llave primaria de la entidad es la llave primaria de la tabla.
  • Relaciones N:M (Muchos a Muchos):
    • Se crea una NUEVA TABLA (tabla intermedia o asociativa) para representar la relación.
    • Esta nueva tabla tendrá como columnas las llaves primarias de AMBAS entidades que relaciona. Estas dos llaves juntas usualmente forman la llave primaria de la nueva tabla.
    • Si la relación E-R tenía atributos, estos se convierten en columnas en esta nueva tabla.
    • Ejemplo: ESTUDIANTES (id_estudiante, nombre) y MATERIAS (id_materia, nombre_materia). La relación N:N "Estudiante_Cursa_Materia" se vuelve una tabla INSCRIPCIONES (id_estudiante_fk, id_materia_fk, fecha_inscripcion).
  • Relaciones 1:N (Uno a Muchos):
    • Opción preferida: Se añade la llave primaria de la entidad del lado "1" como una llave foránea (FK) en la tabla de la entidad del lado "N".
    • Los atributos de la relación (si los hay) se añaden como columnas en la tabla del lado "N".
    • Ejemplo: CLIENTE (id_cliente, nombre) y PEDIDOS (id_pedido, fecha). Un cliente puede tener muchos pedidos. En la tabla PEDIDOS, se añade `id_cliente_fk`.
    • Alternativa (menos común para 1:N puras): Crear una tabla intermedia como en N:M. Esto se considera si hay muchos atributos en la relación, o si se espera que la relación pueda volverse N:M en el futuro, o si la participación del lado "N" es opcional y podría generar muchos nulos.
  • Relaciones 1:1 (Uno a Uno):
    • Hay varias formas, depende de la "participación" (si es obligatorio que existan ambas entidades en la relación):
      • Si ambas participaciones son (1,1) (obligatorias): Puedes fusionar las dos entidades en una sola tabla, o elegir una tabla y añadir la PK de la otra como FK (y UNIQUE). La elección depende de la frecuencia de acceso.
      • Si una es (1,1) y la otra (0,1) (opcional): Se propaga la PK de la entidad (1,1) a la tabla de la entidad (0,1) como FK (y UNIQUE).
      • Si ambas son (0,1) (opcionales): Es mejor crear una tabla separada para la relación (como en N:M pero con PKs únicas) para evitar muchos nulos si las entidades no están relacionadas.
  • Atributos Multivaluados:
    • Se crea una NUEVA TABLA para el atributo multivaluado.
    • Esta nueva tabla tendrá una columna para el atributo en sí y una columna con la llave primaria de la entidad original (como FK).
    • Ejemplo: EMPLEADO (id_empleado, nombre) tiene un atributo multivaluado "Telefonos". Se crea TELEFONOS_EMPLEADO (id_empleado_fk, numero_telefono). La PK aquí sería la combinación de ambas.
  • Relaciones Ternarias (Grado 3):
    • Muchos a Muchos a Muchos (N:M:P): Se crea una nueva tabla. Su PK es la concatenación de las PKs de las 3 entidades participantes. Los atributos de la relación se añaden a esta tabla.
    • Muchos a Muchos a Uno (N:M:1): Se crea una nueva tabla. Su PK es la concatenación de las PKs de las entidades de los lados N y M. La PK de la entidad del lado '1' se añade como FK. Atributos de la relación van a esta tabla.

Conceptos del Modelo Relacional (¡Tabla de equivalencias!)

Es útil saber cómo se llaman las cosas en el mundo relacional y cómo se comparan con otros contextos:

Terminología Relacional Equivalente en "Tablas" Equivalente en "Archivos" (más antiguo) Explicación Sencilla
Relación Tabla Archivo La estructura principal donde se guardan los datos, como una hoja de cálculo.
Tupla Fila / Renglón Registro Una instancia completa de datos, una "fila" en la tabla. Ej: los datos de UN estudiante.
Atributo Columna / Campo Campo Una característica específica de los datos. Ej: "Nombre", "Edad".
Dominio Tipo de dato + Restricciones - El conjunto de valores válidos para un atributo. Ej: para "Edad", números enteros entre 0 y 120.
Esquema de la Relación Definición de la Tabla - El nombre de la tabla y la lista de sus columnas. Ej: ALUMNOS(Matricula, Nombre, Carrera).
Llave Primaria (PK) Llave Primaria Llave Primaria Columna(s) que identifican unívocamente cada fila.
Llave Foránea (FK) Llave Foránea - Columna(s) en una tabla que hacen referencia a la PK de otra tabla, estableciendo una relación.
Grado (de la Relación) Número de Columnas Número de Campos Cuántos atributos (columnas) tiene una tabla.
Cardinalidad (de la Relación) Número de Filas Número de Registros Cuántas tuplas (filas) tiene una tabla en un momento dado.

Dato curioso: El término "Relacional" no viene de "relaciones entre tablas", sino del concepto matemático de "Relación" que es, en esencia, un conjunto de tuplas (una tabla).

Tipos de Datos Comunes en SQL

SQL soporta muchos tipos de datos, algunos comunes son:

  • Numéricos: INT (enteros), DECIMAL, FLOAT (números con decimales).
  • Texto: CHAR (fijo), VARCHAR (variable), TEXT (grande).
  • Fecha/Hora: DATE, TIME, TIMESTAMP.
  • Booleanos: BOOLEAN (TRUE/FALSE).
  • Otros: BINARY, BLOB (para archivos), JSON, XML.

SQL Básico para crear estructuras:


-- Crear una base de datos
CREATE DATABASE mi_escuela_utm;

-- Seleccionar la base de datos para usarla
USE mi_escuela_utm;

-- Crear una tabla llamada Alumnos
CREATE TABLE Alumnos (
    id_alumno INT PRIMARY KEY,       -- Llave primaria, tipo entero
    nombre VARCHAR(50),             -- Texto de hasta 50 caracteres
    apellido_paterno VARCHAR(50),
    fecha_nacimiento DATE,
    carrera_id INT,                  -- Esto sería una llave foránea
    FOREIGN KEY (carrera_id) REFERENCES Carreras(id_carrera) -- Ejemplo de FK
);

-- Crear una tabla Carreras (necesaria para el ejemplo anterior)
CREATE TABLE Carreras (
    id_carrera INT PRIMARY KEY,
    nombre_carrera VARCHAR(100) UNIQUE -- Nombre único para la carrera
);

La Normalización (no detallada en este OCR pero mencionada como objetivo de la unidad) es un proceso para organizar las columnas y tablas de una base de datos relacional para minimizar la redundancia de datos y mejorar la integridad. Se basa en "Formas Normales" (1FN, 2FN, 3FN, BCNF, etc.).
